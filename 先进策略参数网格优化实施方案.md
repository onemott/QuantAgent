# 先进策略参数网格优化实施方案

## 1. 背景与目标

### 1.1 现状痛点

当前的量化回测引擎 `_run_backtest_engine`（位于 `strategy.py`）采用 Python 原生 `for` 循环逐行遍历 K 线数据进行逻辑判断与资金结算。同时，前端交互与可视化存在显著短板。

- **核心代码瓶颈**：
  ```python
  # 典型的解释器开销密集型操作
  for i in range(len(df)):
      # 逐行获取数据，触发大量 PyObject 创建与销毁
      # 动态类型检查导致无法利用 CPU 分支预测与流水线
      if signal == 1: ...
  ```
- **交互与可视化痛点**（新增）：
  - **前端交互缺失**：`strategies/page.tsx` 虽然获取了策略参数元数据，但未提供“参数范围配置”组件。用户无法定义 `Start`, `End`, `Step`，导致 `/api/v1/strategy/optimize` 实际上仅执行了默认参数的单次运行，失去了网格搜索的意义。
  - **结果展示单一**：仅以表格形式展示 Top 10 结果。缺乏**参数敏感度分析**（如热力图），用户无法判断当前最优参数是“孤岛”（过拟合）还是处于“高原”（稳健区间）。
- **性能影响**：
  - 单次日线级别回测耗时在 100ms-500ms 级别。
  - **参数网格优化场景**：若有 3 个参数，每个参数 10 个步长，组合数为 $10^3=1000$ 次回测。总耗时可能达到 100s-500s，用户体验极差。
  - **生产级要求**：生产环境通常需要分钟级甚至 Tick 级数据，数据量激增 100-1000 倍，现有架构完全不可用。

### 1.2 优化目标

构建**分层级、高性能、可扩展**的下一代回测架构：

1. **极速筛选（L1）**：利用向量化技术，将单次核心回测耗时压缩至 **1ms** 以内（提升 100x+）。
2. **精确模拟（L2）**：利用 JIT 技术，在保留复杂逻辑（止损、资金管理）的同时，接近 C++ 原生性能（提升 50x+）。
3. **并发吞吐（L3）**：利用多核并行计算，线性提升参数扫描效率。

***

## 2. 技术方案选型与架构设计

我们采用 **"Vectorization First, Numba Second, Parallelism Always"** 的混合架构策略。

### 2.1 方案一：全向量化回测（Vectorized Backtesting）—— 快速初筛层

**原理**：利用 `numpy` / `pandas` 的列式计算能力，利用 CPU 的 SIMD（单指令多数据）指令集，消除 Python 循环开销。

- **适用场景**：
  - 信号生成逻辑不依赖当前持仓状态。
  - 固定比例/手数下单，无复杂资金管理。
  - 参数大范围粗略扫描。
- **核心逻辑重构**：
  ```python
  # 伪代码示例：消除 for 循环
  df['returns'] = df['close'].pct_change()
  df['strategy_returns'] = df['signal'].shift(1) * df['returns']
  # 扣除成本
  df['strategy_returns'] -= abs(df['signal'].diff()) * commission_rate
  # 计算累计净值
  equity_curve = (1 + df['strategy_returns']).cumprod()
  ```

### 2.2 方案二：Numba JIT 编译（Just-In-Time）—— 精确回测层

**原理**：使用 `numba` 装饰器将 Python 函数即时编译为 LLVM 机器码。

- **适用场景**：
  - 策略逻辑包含路径依赖（如：移动止损需根据上一步的最高价计算）。
  - 复杂的资金管理（如：基于当前账户权益计算开仓手数）。
  - 无法向量化的逻辑分支。
- **核心逻辑重构**：
  ```python
  from numba import njit, float64, int64

  @njit(float64[:](float64[:], int64[:], float64, float64), cache=True)
  def fast_backtest_core(prices, signals, initial_capital, commission):
      n = len(prices)
      equity = np.zeros(n)
      cash = initial_capital
      position = 0.0
      
      for i in range(n):
          # 编译后的机器码循环，速度接近 C/C++
          if signals[i] == 1 and position == 0:
              # 开仓逻辑...
          elif signals[i] == -1 and position > 0:
              # 平仓逻辑...
          equity[i] = cash + position * prices[i]
      return equity
  ```

### 2.3 方案三：多进程并行（Multiprocessing）—— 算力横向扩展

**原理**：Python GIL 锁限制了多线程计算密集型任务的效率，需采用多进程（ProcessPool）绕过 GIL，利用多核 CPU。

- **实施策略**：
  - 使用 `concurrent.futures.ProcessPoolExecutor`。
  - **数据共享优化**：使用 `SharedMemory` 或在主进程预处理数据后只传递必要参数（如 numpy 数组引用），避免进程间大量数据序列化/反序列化（Pickle）的开销。

### 2.4 方案四：前端交互与可视化增强 (Interaction & Visualization)

**目标**：解决“难用”的问题，提供生产级的参数调优体验。

1. **动态参数配置组件 (Dynamic Parameter Configurator)**：
   - **实现机制**：在 `strategies/page.tsx` 中，根据选定策略的 `params` 元数据（Min/Max/Default），动态渲染输入控件。
   - **交互逻辑**：由单一数值输入改为 **“范围输入模式”**：
     - `Start` (起始值)
     - `End` (结束值)
     - `Step` (步长)
   - **安全拦截**：前端实时计算 `Total Combos = Π ( (End - Start) / Step + 1 )`。若总组合数 > 500，显示警告；若 > 1000，禁用“开始优化”按钮，防止浏览器卡死或后端超时。
2. **多维结果可视化 (Visual Analytics)**：
   - **2D 热力图 (Heatmap)**：
     - *适用场景*：双参数策略（如 MA 的 `fast_period` vs `slow_period`）。
     - *实现*：X轴为参数A，Y轴为参数B，色块颜色深浅代表夏普比率或年化收益。帮助用户一眼识别“参数高原”（稳健区域）。
   - **平行坐标图 (Parallel Coordinates)**：
     - *适用场景*：多参数策略 (>2)。
     - *实现*：每根纵轴代表一个参数或指标，线条连接一次回测的所有属性。高亮显示 Top 10% 的线条，观察高性能参数的分布规律。

### 2.5 方案五：智能搜索算法 (Heuristic Search) —— 解决维数灾难

**原理**：当参数维度增加（>3）时，网格搜索的计算量呈指数级爆炸。智能搜索算法通过“探索与利用”（Exploration vs Exploitation）策略，在参数空间中高效寻找全局最优解，而非遍历所有点。

- **核心算法选型**：
  1. **随机搜索 (Random Search)**：
     - *机制*：在参数空间内随机采样 N 次。
     - *优势*：在同等计算预算下，比网格搜索更有可能找到重要参数的最优值（Bergstra & Bengio, 2012）。
  2. **遗传算法 (Genetic Algorithm, GA)**：
     - *机制*：模拟自然选择，通过交叉（Crossover）、变异（Mutation）和选择（Selection）迭代进化参数种群。
     - *优势*：适合非凸、多峰值的复杂目标函数，不易陷入局部最优。
     - *库选型*：`deap` 或 `scikit-opt`。
  3. **贝叶斯优化 (Bayesian Optimization)**：
     - *机制*：构建目标函数的概率模型（高斯过程），利用采集函数（Acquisition Function）指导下一步采样点。
     - *优势*：样本效率极高，适合单次回测成本昂贵的场景。
     - *库选型*：`optuna` (推荐) 或 `hyperopt`。
- **实施策略**：
  - 引入 `optuna` 库作为核心优化器。
  - 定义 `objective(trial)` 函数，内部调用向量化/Numba 回测引擎。
  - 支持 `Pruning`（剪枝）：如果在回测早期发现表现极差，直接终止该次尝试。

## 3. 落地实施路线图

### 阶段一：向量化引擎实现 (v2.0)

**目标**：解决 90% 的常规参数优化需求，提供秒级响应。

1. **数据预处理标准化**：
   - 将 `strategy.py` 中的 `run_backtest` 输入从 DataFrame 逐步过渡到 Numpy Array。
   - 预计算常用指标（OHLC、Returns）。
2. **实现** **`VectorizedBacktester`** **类**：
   - 仅支持 `signal` 信号矩阵输入。
   - 输出 `equity_curve` 和基础统计指标（Sharpe, MaxDD）。
3. **集成点**：
   - 在参数网格搜索模块中，优先调用向量化引擎。

### 阶段二：Numba 核心加速 (v2.1)

**目标**：处理“特定路径依赖”策略的优化。

1. **提取核心循环**：
   - 将 `_run_backtest_engine` 中的 `for` 循环逻辑剥离为独立纯函数。
   - 移除所有 pandas 依赖，仅接受 numpy 数组。
2. **Numba 适配**：
   - 添加 `@njit` 装饰器。
   - 处理类型签名，确保 `nopython=True` 模式编译成功。
3. **混合调用**：
   - 对于简单逻辑走向量化。
   - 对于复杂逻辑走 Numba 函数。

### 阶段三：并行计算框架 (v2.2)

**目标**：充分利用服务器/工作站多核性能。

1. **任务分发器**：
   - 将参数网格拆分为任务块（Chunking）。
2. **进程池管理**：
   - 动态调整 worker 数量（通常为 CPU 核数 - 1）。
3. **结果聚合**：
   - 异步收集回测结果，实时更新前端进度条。

### 阶段四：智能搜索集成 (v2.3)

**目标**：支持多参数复杂策略的高效寻优。

1. **集成 Optuna**：
   - 添加 `optuna` 依赖。
   - 封装 `OptunaOptimizer` 类。
2. **接口适配**：
   - 前端支持选择“优化算法”：`Grid Search` vs `TPE (Bayesian)` vs `Random`。
   - 后端 `/optimize` 接口新增 `algorithm` 和 `n_trials` 参数。

将这五种方案看作一个金字塔架构的四个不同层级，它们各司其职：

- 交互层 (User Interface) —— 方案四 (前端交互)
  - 作用 ：这是用户接触系统的入口。无论底层多快，如果前端不能配置参数范围、看不到热力图，系统就是“不可用”的。
  - 地位 ：必选项，与其他所有方案共存。
- 调度层 (Orchestration) —— 方案五 (智能搜索) vs 朴素网格
  - 作用 ：决定“测哪些参数”。
  - 共存方式 ：系统可以提供两种模式供用户选择。
    - 小规模探索 ：使用全量网格搜索（Grid Search）。
    - 大规模寻优 ：当参数组合超过 1000 组时，自动切换或用户手动选择 Optuna (方案五) 算法。
- 并发层 (Concurrency) —— 方案三 (多进程)
  - 作用 ：利用多核 CPU。
  - 共存方式 ：它是一个“外壳”。无论内部跑的是向量化引擎还是 Numba 引擎，都可以被包裹在 ProcessPoolExecutor 中进行并行分发。
- 核心计算层 (Core Engine) —— 方案一 (向量化) & 方案二 (Numba)
  - 作用 ：执行单次具体的策略回测。
  - 共存方式 ：这是最需要“取舍”的地方，但也可以 混合使用 。
    - 场景 A ：简单的指标交叉（如 MA, RSI），直接用 pandas/numpy 向量化 (方案一) ，速度最快，代码最简洁。
    - 场景 B ：复杂的路径依赖（如移动止损、基于净值的加仓），向量化写不出来，此时自动降级使用 Numba (方案二) 编译的循环逻辑。
