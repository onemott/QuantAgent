目前的系统处于“**原型验证（PoC）完成，工程化初期**”的阶段。
- **已完成**：Redis 基础集成、基础风控引擎 (`RiskManager`)、初步的回测脚本。
- **待解决**：**伪实时数据流**（仍依赖轮询）、**无状态 Agent**、**不支持做空**、**任务调度缺失**、**缺乏订单状态管理**。

### **🚀 全栈量化系统优化实施方案 (v2.7)**

#### **一、 数据链路重构：从“伪实时”到“金融级流式处理”**

**当前痛点**：
- **伪实时与同步阻塞**：`backend/main.py` 仍使用 `while True` 循环每秒轮询，且 `binance_service.py` 内部使用 `run_in_executor` 包装同步的 `ccxt` 库，导致大量线程切换开销，延迟高 (>1000ms)。
- **双轨制混乱**：`gateway/` (Go) 已实现 WebSocket -> NATS 的高效链路，但 Python 后端尚未接入，仍运行低效的轮询代码。
- **缺乏并发保护**：Redis 读写采用 JSON 序列化（性能低），且缺乏分布式锁，高并发下存在数据竞争风险。

**🛠 优化方案**：

1.  **接入高可用 WebSocket 长连接 (两种路径)**
    -   **路径 A (推荐 - 架构统一)**：Python 后端接入 **NATS** 总线，直接消费由 Go Gateway 发布的标准化行情数据，复用现有的高性能 Go 微服务。
    -   **路径 B (独立 - 快速迭代)**：引入 `binance-connector` (Async) 或 `ccxt.pro`，在 Python 端建立独立 WebSocket 连接。
    -   **关键技术点**：
        -   **心跳与重连**：实现指数退避 (Exponential Backoff) 重连机制。
        -   **数据完整性 (Gap Recovery)**：利用 Update ID 校验连续性，丢包时自动降级 REST API 补全。

2.  **全链路异步化改造 (Async I/O)**
    -   **行动**：彻底移除 `run_in_executor` + 同步 `ccxt` 的低效模式。
    -   **实现**：全面切换至 `ccxt.async_support`，实现原生 `await` 调用，释放事件循环压力。
    -   **事件循环优化**：
        -   **Linux (生产)**: 启用 `uvloop` 替换标准 `asyncio` 循环，吞吐量提升 2-4 倍。
        -   **Windows (开发)**: 保持默认 `ProactorEventLoop` 或使用 `winloop` (**注意：`uvloop` 不支持 Windows，切勿在开发环境强制引入**)。

3.  **Redis 性能与并发优化**
    -   **序列化升级**：使用 **MsgPack** (`msgpack-python`) 代替 `JSON`，体积减小 30%-50%，编解码速度提升 5-10 倍。
    -   **抗并发设计**：引入 **Redis 分布式锁** (基于 `redis-py` 的 Lock 或 Redlock)，在订单状态更新、资金划转等关键临界区防止竞态条件 (Race Condition)。
    -   **架构调整**：拆分 `Ingestion Worker` (写入) 与 `API Server` (读取)，通过 Pub/Sub 解耦。

4.  **引入任务调度系统 (APScheduler)**
    -   **行动**：引入 `APScheduler` (使用 `AsyncIOScheduler`)，新建 `backend/scheduler.py`。
    -   **持久化**：配置 Postgres/Redis 作为 JobStore。
    -   **任务**：ETL 清洗、策略寻优、日报生成。

5.  **增加系统可观测性 (Observability)**
    -   **行动**：新增 `backend/app/api/health.py`。
    -   **监控指标**：ClickHouse/Postgres 连接池状态、Redis 延迟、Binance 权重、WS 连接数。

#### **二、 策略引擎升级：从“单点规则”到“RAG 驱动的有状态博弈”**

**当前痛点**：
- **记忆缺失**：`AgentMemory` 仅为基于 `Redis` 的简单滑动窗口（最近 5 条文本），系统“健忘”，无法从历史教训中学习。
- **逻辑浅薄**：LLM 仅作为翻译官，缺乏对历史相似行情的“直觉”；无法回答“上次遇到这种顶背离+成交量萎缩时，我开空赚钱了吗？”。
- **状态混乱**：缺乏真正的订单状态机 (OMS)，难以处理部分成交、撤单等复杂场景。

**🛠 专家级优化方案 (RAG & Vector Enhancement)**：

1.  **基础设施：PGVector 向量数据库集成**
    -   **行动**：替换标准 Postgres 镜像，启用 `pgvector` 扩展。
    -   **Docker**：使用 `pgvector/pgvector:pg16` 镜像。
    -   **ORM 适配**：在 `SQLAlchemy` 模型中引入 `mapped_column(Vector(1536))` (适配 OpenAI text-embedding-3-small) 或 `768` (适配 HuggingFace 本地模型)。
    -   **索引优化**：为向量列创建 `HNSW` 索引 (Hierarchical Navigable Small World)，实现毫秒级近似最近邻搜索 (ANN)。

2.  **Schema 进化：构建“市场状态-决策-结果”闭环**
    -   **重构 `AgentMemory` 模型**：
        -   `market_state_embedding` (Vector): 核心字段。将 K 线形态（如 RSI, MACD, Bollinger）、宏观情绪、OrderBook 偏斜度拼接后进行 Embedding。
        -   `reasoning_summary` (Text): 当时 Agent 的决策逻辑摘要。
        -   `action` (Enum): BUY / SELL / HOLD。
        -   `outcome_pnl` (Float): **关键新增**。该决策在随后 N 周期内的实际盈亏（PnL）。需通过异步任务（回溯标记）填充。
    -   **数据流闭环**：
        1.  **决策时**：Embedding 当前市场状态 -> 存入 DB (Outcome 为空)。
        2.  **结算时**：T+N 周期后，Scheduler 触发任务，计算该次决策的 PnL，回写 `outcome_pnl`。

3.  **实现 Context-Aware Trading (RAG 推理流)**
    -   **检索策略 (Hybrid Search)**：
        -   在生成新信号前，先进行**混合检索**：
            -   *向量相似度*：找“长得像”的历史行情 (Cosine Distance < 0.2)。
            -   *元数据过滤*：限制同标的 (Symbol)、同时间窗口 (Timeframe)。
    -   **Prompt 增强**：
        -   将检索到的 Top-3 历史案例注入 Prompt：
            > "History Lesson: In 3 similar market situations (High volatility + RSI > 80), you Shorted 2 times and profited 15%, but Longed once and lost 5%. Consider this bias."
    -   **效果**：让 Agent 具备“肌肉记忆”，自动规避历史上的重复错误。

4.  **构建最小化订单管理系统 (Mini-OMS)**
    -   **行动**：在 `PaperTradingService` 中实现完整的状态机。
    -   **状态流转**：`PENDING_NEW` -> `NEW` -> `PARTIALLY_FILLED` -> `FILLED` / `CANCELED` / `REJECTED`。
    -   **幂等性**：确保所有下单请求具备唯一 `client_order_id`，防止网络超时导致的重复下单。

#### **三、 回测系统完善：从“玩具模型”到“实战模拟”**

**当前痛点**：
-   **成交过于理想**：`strategy_backtest.py` 是基于向量化的简单回测，使用当前 K 线 `Close` 价成交，存在严重的**前视偏差 (Look-ahead Bias)**。
-   **缺乏做空机制**：目前代码 `PaperTrade` 表虽有 `side` 字段，但业务逻辑层 (`PaperTradingService`) 不支持做空（Short）开仓。
-   **成本忽略**：未计算资金费率、Maker/Taker 手续费差异及市场冲击成本。

**🛠 优化方案**：

1.  **事件驱动撮合引擎 (Event-Driven Matching Engine)**
    -   **行动**：重写回测逻辑，引入简易撮合引擎。
    -   **规则细化**：
        -   **防前视偏差**：市价单 (Market Order) 必须在信号产生的**下一根 K 线 Open 价**成交，而非当前 Close 价。
        -   **限价单撮合**：基于 K 线 High/Low 穿透逻辑。保守模式下，需满足 `Low < Limit_Price < High` 且假设成交量不超过当根 K 线 Volume 的 10%。
        -   **延迟模拟**：增加网络延迟 (Latency) 模拟，随机增加 50-200ms 的撮合延迟。

2.  **高级滑点与市场冲击模型 (Advanced Slippage & Market Impact)**
    -   **动态滑点**：摒弃固定比例滑点，采用 **ATR (平均真实波幅)** 动态估算波动率滑点。
    -   **市场冲击 (Square Root Law)**：引入经典的交易成本模型 $Impact = \sigma \times \sqrt{\frac{Order Size}{Daily Volume}}$，模拟大额订单对价格的推动。
    -   **订单簿模拟**：若有 Level 2 数据，基于 **Order Book Imbalance (OBI)** 调整成交均价。

3.  **交易成本分析 (TCA - Transaction Cost Analysis)**
    -   **行动**：新增 TCA 分析模块，量化隐性成本。
    -   **核心指标**：
        -   **执行落差 (Implementation Shortfall)**: `| 决策价格 - 实际成交均价 |`。
        -   **资金费率拖累 (Funding Drag)**: 统计长期持仓的资金费率损耗。
        -   **滑点分布**：统计不同波动率环境下的滑点分布情况。

4.  **支持双向交易与费率结构**
    -   **行动**：在 `PaperTradingService` 中完善 `short` 逻辑及费率计算。
    -   **逻辑**：
        -   **做空机制**: 借币卖出 -> 买入还币；计算 `Unrealized PnL` 时需注意方向。
        -   **费率模型**: 区分 Maker (-0.005% ~ 0.02%) 与 Taker (0.02% ~ 0.05%) 费率，并支持 BNB 折扣模拟。

#### **四、 金融级安全与风控体系 (Financial-Grade Security & Risk Control)**

**当前痛点**：
- **密钥裸奔**：依赖 `.env` 明文存储。
- **风控现状**：`RiskManager` 已实现，但**缺少针对“做空”的风控规则**，且缺乏杠杆与保证金管理。
- **缺乏熔断**：没有系统级的“紧急刹车”机制，仅有被动的拒绝开仓。
- **单点防御**：缺乏针对“胖手指”或异常市场波动的动态防御。

**🛠 优化方案**：

1.  **零信任密钥管理**
    -   **行动**：生产环境严禁本地 `.env`，接入 AWS KMS 或 HashiCorp Vault。
    -   **内存安全**：密钥在使用后应尽快从内存中清除，避免 Core Dump 泄露。

2.  **升级风控引擎 (Risk Manager v2 - FRM Standard)**
    -   **行动**：修改 `backend/app/services/risk_manager.py`，引入多层级风控。
    -   **新增规则**：
        -   **全局熔断 (Kill Switch)**：
            -   **机制**：Redis 原子计数器监控全系统盈亏速度。
            -   **触发**：若 1 分钟内回撤 > 2% 或 总权益 < 警戒线，立即触发。
            -   **动作**：Cancel All Orders (撤单) -> Stop All Agents (停机) -> Flatten Positions (可选平仓/对冲)。
        -   **杠杆与保证金 (Leverage & Margin)**：
            -   **动态保证金**：实现阶梯保证金制度 (Tiered Margin)，持仓越大，维持保证金率 (MMR) 越高。
            -   **真实杠杆监控**：实时计算 `Effective Leverage = Total Notional / Equity`，超过 3x (可配) 强制减仓。
        -   **做空风控 (Short Selling Control)**：
            -   **融券模拟**：开空前检查“借币额度” (Locate)，模拟“难借资产” (Hard-to-Borrow) 场景。
            -   **逼空保护 (Short Squeeze Guard)**：若空单亏损速度超过阈值 (如 5%/min)，触发强制止损，防止无限亏损。
        -   **异常交易拦截 (Fat Finger & Anomalies)**：
            -   **价格偏离**：买单价 > 市场价 5% 或 卖单价 < 市场价 5% 时拒绝。
            -   **大单拆分**：单笔订单名义价值 > 日均成交量 (ADV) 的 1% 时拒绝或强制拆单 (TWAP)。
        -   **相关性风控**：防止多个 Agent 同时做多高度相关资产 (如 BTC & ETH)，导致风险敞口叠加。

3.  **全链路审计日志 (Audit Logging)**
    -   **行动**：建立 `audit_logs` 表（不可变 append-only）。
    -   **范围**：不仅仅是交易，还包括配置修改、Agent 启停、人工干预等所有操作。

**架构师备注**：
代码库中 `RiskManager` 基础已经打好，这是一个很好的起点。
**优先级建议**：
1.  **WebSocket 改造**是重中之重，没有低延迟数据，量化系统无从谈起。
2.  **OMS 状态机**是系统稳定性的基石，避免在网络不稳定时出现“幽灵单”或重复下单。
3.  **资金费率回测**对于合约交易至关重要，忽略它会导致回测结果严重失真。
